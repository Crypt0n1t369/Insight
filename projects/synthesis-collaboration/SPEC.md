## 0. Executive Summary

A Telegram-native **collaborative intelligence platform** that applies Karpathy-style persistent compounding synthesis to group conversations. Contributors feed inputs → the system (powered by OpenClaw agents) builds an incrementally maintained shared knowledge space → cross-links and contradictions are surfaced automatically → a structured synthesis is generated → next steps are proposed.

**Key insight (from Karpathy):** Unlike RAG (which rediscoveries knowledge from scratch per query), this platform **compiles knowledge once** and keeps it current. Cross-references exist before questions are asked. Contradictions are flagged before synthesis. The wiki is the artifact.

**Backend:** Built on **OpenClaw** — Kristaps already has it running. The Telegram bot(s) are the interface layer; OpenClaw agents do all the synthesis work, wiki maintenance, and insight cross-linking.

## 1. Concept & Vision

### What This Platform Does
A Telegram-based collaborative intelligence engine that transforms group conversations into a **persistent, compounding knowledge artifact** — not a one-shot answer, but a shared wiki that grows with every contribution. Think of it as a Delphi Method + Karpathy Wiki + group intelligence, delivered through Telegram.

### Why It Matters
Most group wisdom is lost to chat scroll. Even "successful" brainstorms produce a document that no one updates. This platform makes collective intelligence **tangible, persistent, and actionable** — and it gets richer with every interaction, not just one synthesis at the end.

### Core Principle (Karpathy Pattern)
> "The wiki is a persistent, compounding artifact. The cross-references are already there. The contradictions have already been flagged. The synthesis already reflects everything you've read. The wiki keeps getting richer with every source you add and every question you ask."

Applied to group collaboration: every contribution updates the shared knowledge space. Follow-up questions are generated by cross-referencing existing inputs. Syntheses aren't one-shot — they reference an already-cross-linked knowledge artifact.

### Guiding Rules
- Start with 2-3 core functions that work flawlessly
- Add capabilities incrementally
- Autonomy is the goal — minimal human babysitting
- OpenClaw is the brain; Telegram is the face

## 2. The 9-Step Process Flow

*This is the canonical user journey. Every step must be reflected in the bot commands, data model, and state machine.*

```
┌─────────────────────────────────────────────────────────────────────┐
│  STEP 1: LOGIN                                                     │
│  User authenticates via Telegram (link their account)              │
│  → Creates or resumes a persistent User profile                   │
└────────────────────────────┬──────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────────┐
│  STEP 2: CREATE A PROJECT                                          │
│  /new-project <name> [<description>]                               │
│  → Creates a Project in DB, assigns Telegram group as shared space │
│  → Initializes a Session for this topic/challenge                 │
└────────────────────────────┬──────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────────┐
│  STEP 3: "IS THERE A PROBLEM?"                                     │
│  /define-challenge <text>                                         │
│  → Stored as the Project's `challenge_statement`                  │
│  → OpenClaw agent reads it and prepares the problem frame         │
│  → All future contributions are tagged against this challenge      │
└────────────────────────────┬──────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────────┐
│  STEP 4: SHARED SPACE (all inputs visible to all)                  │
│  /insight <text>   OR   forward messages to bot                    │
│  → Stored as Contribution, linked to Project + Session              │
│  → OpenClaw: extracts key claims/entities, updates shared wiki      │
│  → Bot confirms: "Added ✓ (insight #N in this session)"            │
│  → Cross-links are generated: "Related to insight #3, #7"         │
└────────────────────────────┬──────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────────┐
│  STEP 5: INDIVIDUAL FOLLOW-UP QUESTIONS (with cross-linking)        │
│  OpenClaw agent analyzes shared inputs, identifies gaps/conflicts │
│  → Spawns 1v1 child bots (Managed Bots API) OR sends private DMs    │
│  → Each user gets targeted questions: "You mentioned X, can you     │
│    elaborate? Does this contradict Y from Insight #3?"              │
│  → Responses feed back into shared space (visible to all if shared) │
│  → Cross-links updated: "User B's follow-up on X connects to A's   │
│    Insight #3"                                                      │
└────────────────────────────┬──────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────────┐
│  STEP 6: PERSONAL SPACE (confirmations per user)                    │
│  User reviews their contributions + follow-up answers              │
│  /confirm <insight-id>  OR  /edit <insight-id> <new-text>         │
│  → Confirmed contributions get a `confirmed: true` flag            │
│  → Bot shows: "Your confirmed insights for this project: N items"  │
│  → Unconfirmed items flagged as "draft" in synthesis               │
└────────────────────────────┬──────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────────┐
│  STEP 7: GENERATE STEP                                             │
│  /generate  OR  auto-trigger after N confirmed contributions       │
│  → OpenClaw agent reads the entire shared wiki for this project    │
│  → LLM synthesizes: themes, commonalities, divergences              │
│  → Contradictions flagged explicitly ("Insight #2 vs #5")          │
│  → Structured outline + next steps produced                        │
└────────────────────────────┬──────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────────┐
│  STEP 8: OUTPUT — SYNTHESIZED PERSPECTIVES                         │
│  Bot posts to group: structured markdown synthesis                  │
│  Format:                                                             │
│  ## 🎯 Challenge: [problem statement]                              │
│  ## 📌 Commonalities (what most agree on)                            │
│  ## ⚡ Divergences (contradictions + why they matter)               │
│  ## 🔗 Cross-links (key connections between insights)              │
│  ## 📋 Structured Outline                                          │
│  ## ➡️ Next Steps (2-3 concrete proposals)                          │
└────────────────────────────┬──────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────────┐
│  STEP 9: NEXT STEP SUGGESTIONS                                      │
│  → Output includes 2-3 next-step proposals                          │
│  → Group votes via reactions (/vote <1-3>)                         │
│  → Top-voted next step → can start new session or project          │
└─────────────────────────────────────────────────────────────────────┘
```

## 3. Bot Architecture

### Overview: OpenClaw-Powered Synthesis
```
Telegram User → Parent Bot (grammY) → OpenClaw Agent (brain) → Shared Wiki (FS/DB)
                                ↓
                         Managed Child Bot (1v1)
```

### The Two-Bot Model
- **Parent Bot** (`@SynthesisParentBot`): handles all group interactions, project management, shared space, `/insight`, `/generate`
- **Child Bot** (Managed Bot, spawned via Bot API 9.6): handles 1v1 follow-up questions, personal space confirmations, private deep dives

### How OpenClaw Powers the Brain
OpenClaw agents are the **synthesis engine**. The Telegram bot(s) are the interface.
1. **Contribution ingestion** → OpenClaw writes extraction notes to the shared wiki (incremental, not batch)
2. **Cross-link generation** → OpenClaw updates `index.md` and adds cross-refs after each contribution
3. **Follow-up question generation** → OpenClaw identifies gaps/conflicts, proposes targeted questions
4. **Synthesis** → OpenClaw reads the full project wiki, runs the synthesis prompt, returns markdown
5. **Lint/health check** → Periodic (or on-demand `/audit`) scans for contradictions, orphans, stale claims

### Managed Bots Flow (Bot API 9.6 — April 2026)
```
Parent Bot                    Telegram              Child Bot
    │                            │                      │
    │──KeyboardButton──────────→│                      │
    │   (request_managed_bot)    │                      │
    │                            │                      │
    │←──ManagedBotCreated────────│                      │
    │   (preliminary token)      │                      │
    │                            │                      │
    │──getManagedBotToken────────→│                      │
    │                            │                      │
    │←──permanent token──────────│                      │
    │                            │                      │
    │ [launch child bot process with token]             │
    │                            │←──User chats 1v1────→│
    │                            │                      │
    │ [DB: store child bot state + contributions]       │
```

### Telegram Bot Commands (MVP)

| Command | Scope | What it does |
|---|---|---|
| `/start` | DM | Welcome + link to group or create new project |
| `/new-project <name>` | DM/Group | Creates project, sets up session |
| `/define-challenge <text>` | Group | Sets the problem statement (Step 3) |
| `/insight <text>` | Group | Adds contribution to shared space (Step 4) |
| `/my-insights` | DM | Personal space: user's contributions + confirmations |
| `/confirm <id>` | DM | Confirm an insight (Step 6) |
| `/generate` | Group | Triggers OpenClaw synthesis → posts result (Step 7-8) |
| `/status` | Group | Shows project health: N contributions, N confirmed, N follow-ups pending |
| `/followup` | DM | Answer follow-up question from OpenClaw |
| `/vote <1-3>` | Group | Vote on next step proposal |
| `/audit` | Group | Run wiki health check (Karpathy lint pattern) |
| `/help` | Any | Command reference |

### Group Types
- Groups and Supergroups both supported
- Forum topics: optionally route synthesis to a dedicated `#synthesis` thread
- Bot must be admin to see all messages in supergroups (for passive contribution capture)

### Storage
- **SQLite** for MVP (zero-config)
- **PostgreSQL** via Prisma for production
- Wiki pages also stored as actual markdown files in `projects/<id>/wiki/` (Karpathy pattern: the wiki is a git repo of markdown files)
- Bot tokens encrypted with `AES-256-GCM` before storage

## 5. OpenClaw Integration

### Why OpenClaw as Backend
Kristaps already has OpenClaw running. The platform should **build on his existing stack**:
- OpenClaw agents are the LLM-powered brain (synthesis, cross-linking, follow-up generation, wiki maintenance)
- OpenClaw memory system (`memory/`) can store project context and recall
- OpenClaw's subagent spawning handles the parallel 1v1 follow-up flow naturally
- Telegram bot is just the **interface layer** calling OpenClaw tools

### Integration Points

**1. OpenClaw as Synthesis Engine**
```
Telegram /generate
  → Parent bot sends project context + all contributions + wiki pages to OpenClaw agent
  → OpenClaw agent runs synthesis (Karpathy-style: reads wiki, extracts themes, finds contradictions)
  → Returns structured markdown
  → Bot posts to group
```

**2. OpenClaw for Wiki Maintenance (Incremental Ingest)**
```
Telegram /insight
  → Parent bot stores contribution in DB
  → Calls OpenClaw agent: "Ingest this contribution into project wiki. Extract key claims, update index, add cross-links."
  → OpenClaw agent writes wiki pages (in DB or FS)
  → Bot confirms with cross-link note: "Added ✓ — related to Insight #3 (X), Insight #7 (Y)"
```

**3. OpenClaw for Follow-Up Question Generation**
```
On N contributions (configurable threshold)
  → OpenClaw agent: "Analyze current wiki for project X. Identify top 3 gaps or conflicts."
  → For each gap: spawn subagent or send targeted question via child bot
  → Store question/answer, update wiki
```

**4. OpenClaw Subagent per Follow-Up (1v1 Child Bot)**
- Each child bot session is handled by an OpenClaw subagent
- Subagent has project context + user's confirmed insights
- Asks 3-5 targeted questions, writes answers to DB
- On completion: synthesizes user's personal perspective → sends to parent

**5. OpenClaw Memory Integration**
- `memory/01-areas/` — active projects list
- `memory/03-projects/<project-id>/` — per-project wiki and context
- `memory/02-resources/` — shared templates, prompt schemas

### OpenClaw Tool Calls from Bot
```typescript
// Pseudocode — how parent bot calls OpenClaw
async function synthesizeProject(projectId: string): Promise<SynthesisOutput> {
  const wiki = await getWikiPages(projectId);
  const contributions = await getContributions(projectId);
  
  const result = await openclad.runAgent({
    agent: "synthesis",
    context: {
      project_id: projectId,
      challenge: await getChallengeStatement(projectId),
      wiki_pages: wiki,
      contributions: contributions,
    },
    prompt: SYNTHESIS_PROMPT  // see Appendix A
  });
  
  return parseAndStore(result);
}
```

### Karpathy Pattern Alignment

| Karpathy Layer | Synthesis Platform Implementation |
|---|---|
| Raw sources | User contributions (Step 4 inputs) — immutable in DB |
| The wiki | Project wiki pages (OpenClaw-generated markdown + DB records) |
| The schema | OpenClaw agent instructions + `memory/03-projects/<id>/SYSTEM.md` |
| Ingest | `/insight` → OpenClaw extracts → wiki update |
| Query | `/generate` → OpenClaw reads wiki → synthesis |
| Lint | `/audit` → OpenClaw checks for contradictions, orphans, stale claims |
| index.md | Project summary page, auto-updated by OpenClaw |
| log.md | Per-project log of all ingests, queries, generations |

## Appendix A: Synthesis Prompt (OpenClaw)

```markdown
You are maintaining the shared knowledge wiki for project: {{project_name}}
Challenge statement: {{challenge_statement}}

## Your tasks:

### 1. Ingest (on each new contribution)
- Read the new contribution
- Extract key claims, entities, and any contradictions with existing wiki pages
- Update relevant entity/concept pages
- Add cross-links to index.md
- Append entry to log.md

### 2. Synthesize (on /generate)
Read ALL wiki pages for this project. Then produce:

## 🎯 Challenge
[problem statement]

## 📌 Commonalities
[what most contributors agree on — with insight refs]

## ⚡ Divergences
[contradictions or different framings — with insight refs and why they matter]

## 🔗 Cross-links
[key connections between insights that contributors didn't explicitly link]

## 📋 Structured Outline
[headings + bullet points distilling the collective input]

## ➡️ Next Steps
[2-3 concrete proposals, each as: "Do X to achieve Y"]

### 3. Lint (on /audit)
Scan all wiki pages for:
- Contradictions between pages
- Stale claims superseded by newer insights
- Orphan pages with no inbound links
- Missing cross-references
- Suggest new questions to investigate
```
